# 项目协作规则

本文件适用于当前仓库根目录及其所有子目录。后续所有自动化助手、代码代理和文档产出都必须遵守以下规则。

## 一、文档语言规则

- 所有文档型产出必须以中文内容为主，不能出现纯英文说明段落，也不能出现乱码内容。
- 允许保留必要的英文专有名词、文件路径、类名、方法名、字段名、命名空间、插件名和技术名词。
- 如果引用或分析外部资源时遇到编码异常，必须先修复或说明编码问题，不能把乱码直接写入正式文档。
- 文档中的代码块可以包含英文标识符，但代码块外的解释说明必须使用中文。

## 二、Git 操作规则

- 未经用户明确允许，不能私自执行任何 Git 上传、提交或历史修改操作。
- 禁止擅自执行 `git commit`、`git push`、`git pull`、`git merge`、`git rebase`、`git reset`、`git stash`、`git tag`、创建或切换分支等会改变仓库状态或远端状态的操作。
- 如确实需要 Git 操作，必须先说明原因、影响范围和具体命令，等待用户明确确认。
- 允许在需要时读取 Git 信息，例如查看状态、差异、日志或责任行，但不能擅自修改 Git 状态。

## 三、文档归档规则

- 所有项目文档必须写入 `Docs` 文件夹下，不能散落在项目根目录或资源目录中。
- `Docs` 下固定分为三个文档区域：`Docs/总设计文档`、`Docs/程序文档`、`Docs/策划文档`。
- `Docs/总设计文档` 只存放项目总体设计、顶层模块划分、模块之间的连接关系和整体架构说明；不写单个模块内部的具体玩法细节，也不写具体施工步骤。
- `Docs/策划文档` 只存放单个模块的玩法设计、规则设计、内容设计、数值方向、体验目标和策划方案；不写技术实现、代码结构或开发施工清单。
- `Docs/程序文档` 只存放技术拆解、模块施工清单、开发步骤、接口设计、实现方案、依赖关系和验证方式。
- 新建文档前必须先判断文档类型，再放入对应子目录。
