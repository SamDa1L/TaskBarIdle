# 《桌面裂隙》项目目录资产结构

> **文档类型：程序文档**  
> **适用项目：桌面裂隙 / Desktop Rift**  
> **依据文档：《桌面裂隙》项目大纲、《桌面裂隙》游戏设计策划案**  
> **推荐引擎：Godot 4.x**  
> **文档用途：统一项目目录、代码模块、场景资源、数据资产和命名规范**

---

## 1. 结构设计目标

本项目的目录资产结构需要服务“桌面常驻地下城经营防守游戏”的核心形态，重点满足以下目标：

- 支持桌面窗口、地下城模拟、自动战斗、资源成长、事件战报等模块并行开发。
- 区分场景、脚本、配置数据、美术、音频、字体、插件和测试内容，避免资源散落。
- 保持 Godot 项目的场景复用能力，让房间、魔物、冒险者、陷阱、UI 面板都可以独立维护。
- 优先采用数据驱动结构，让魔物、冒险者、房间、陷阱、事件和战报可以通过配置扩展。
- 为 `VS → FVS → MVP → Alpha → CBT → Beta → EA → 上线版本` 的迭代路线预留扩展空间。

## 4. 目录职责说明

### 4.1 `scenes` 场景目录

`scenes` 用于存放所有可实例化的 Godot 场景。项目应优先通过小场景组合形成完整游戏，避免把大量逻辑堆在单一主场景中。

```text
scenes/
├─ desktop/
│  ├─ desktop_root.tscn            # 桌面常驻根场景
│  ├─ rift_window.tscn             # 裂隙窗口主体
│  ├─ collapsed_bar.tscn           # 收起状态条
│  └─ window_debug_panel.tscn      # 窗口调试面板
├─ dungeon/
│  ├─ dungeon_root.tscn            # 地下城根场景
│  ├─ room_slot.tscn               # 房间格
│  ├─ rooms/
│  │  ├─ entrance_room.tscn
│  │  ├─ monster_lair.tscn
│  │  ├─ trap_room.tscn
│  │  ├─ bait_room.tscn
│  │  ├─ production_room.tscn
│  │  └─ core_room.tscn
│  └─ props/                       # 地下城装饰与可交互物
├─ entities/
│  ├─ monsters/
│  ├─ adventurers/
│  └─ drops/
├─ gameplay/
│  ├─ simulation_root.tscn         # 地下城模拟根节点
│  ├─ combat_resolver.tscn         # 自动战斗处理节点
│  └─ offline_result_preview.tscn  # 离线结算预览
├─ ui/
│  ├─ hud/
│  ├─ panels/
│  ├─ popups/
│  └─ reports/
└─ vfx/
   ├─ trap_trigger_vfx.tscn
   ├─ soul_gain_vfx.tscn
   └─ rift_pulse_vfx.tscn
```

场景拆分原则：

- `desktop` 只处理桌面窗口形态、贴边、收起、展开和调试显示。
- `dungeon` 只处理地下城空间结构、房间和核心表现。
- `entities` 只处理魔物、冒险者、掉落物等可复用实体。
- `gameplay` 只承载模拟、战斗、结算等系统型节点。
- `ui` 只处理界面表现和玩家操作入口。
- `vfx` 只存放可复用演出效果，避免散落在业务场景中。

### 4.2 `scripts` 脚本目录

`scripts` 用于存放全部 GDScript。脚本应按系统职责拆分，优先使用信号和数据资源解耦，不让 UI、战斗和数据配置互相硬编码。

```text
scripts/
├─ autoload/
│  ├─ game.gd                     # 游戏全局状态入口
│  ├─ signal_bus.gd               # 全局信号转发
│  ├─ data_registry.gd            # 数据加载与索引
│  ├─ save_manager.gd             # 存档管理
│  ├─ audio_manager.gd            # 音频管理
│  └─ desktop_window_manager.gd   # 桌面窗口全局控制
├─ desktop/
│  ├─ desktop_root.gd
│  ├─ rift_window.gd
│  ├─ window_snapper.gd
│  ├─ window_mode_controller.gd
│  └─ mouse_passthrough_controller.gd
├─ dungeon/
│  ├─ dungeon_model.gd
│  ├─ dungeon_controller.gd
│  ├─ room_slot.gd
│  ├─ room_instance.gd
│  └─ core_controller.gd
├─ entities/
│  ├─ actor.gd                    # 魔物和冒险者共用基础实体
│  ├─ monsters/
│  ├─ adventurers/
│  └─ drops/
├─ gameplay/
│  ├─ invasion_director.gd        # 入侵生成与节奏
│  ├─ combat_resolver.gd          # 自动战斗结算
│  ├─ trap_resolver.gd            # 陷阱触发与反制
│  ├─ resource_economy.gd         # 资源产出和消耗
│  ├─ progression_controller.gd   # 成长与解锁
│  ├─ event_director.gd           # 事件触发
│  └─ report_generator.gd         # 战报生成
├─ resources/
│  ├─ monster_data.gd
│  ├─ adventurer_data.gd
│  ├─ room_data.gd
│  ├─ trap_data.gd
│  ├─ event_data.gd
│  └─ resource_cost_data.gd
├─ save/
│  ├─ save_data.gd
│  └─ save_migration.gd
├─ ui/
│  ├─ hud_controller.gd
│  ├─ management_panel.gd
│  ├─ room_build_panel.gd
│  ├─ monster_panel.gd
│  └─ battle_report_panel.gd
└─ utils/
   ├─ math_utils.gd
   ├─ text_template.gd
   └─ weighted_table.gd
```

脚本拆分原则：

- `autoload` 只放真正全局的服务，不放具体玩法细节。
- `resources` 中的脚本只定义数据结构，不写复杂业务逻辑。
- `gameplay` 负责系统规则，`ui` 负责显示和输入，两者通过信号或控制器通信。
- `desktop` 与玩法系统保持边界，窗口模式变化不应直接修改战斗或经济逻辑。
- 高频生成对象应预留对象池或复用机制，避免桌面常驻时产生不稳定性能开销。

### 4.4 `assets` 资产目录

`assets` 用于存放美术、音频、字体和 Shader 等内容资产。导入 Godot 后生成的 `.import` 文件由引擎维护，不应手动编辑。

```text
assets/
├─ art/
│  ├─ characters/
│  │  ├─ monsters/
│  │  └─ adventurers/
│  ├─ dungeon/
│  │  ├─ rooms/
│  │  ├─ props/
│  │  └─ tilesets/
│  ├─ desktop/
│  │  ├─ rift/
│  │  ├─ window_frame/
│  │  └─ tray_icon/
│  ├─ ui/
│  │  ├─ icons/
│  │  ├─ panels/
│  │  └─ buttons/
│  └─ vfx/
├─ audio/
│  ├─ music/
│  ├─ sfx/
│  │  ├─ ui/
│  │  ├─ combat/
│  │  ├─ traps/
│  │  └─ desktop/
│  └─ ambience/
├─ fonts/
└─ shaders/
```

资产拆分原则：

- 魔物、冒险者和房间资源按玩法对象归档，方便数据和场景引用。
- UI 图标按资源、房间、魔物、陷阱、状态等用途继续细分。
- 桌面窗口相关素材独立放入 `assets/art/desktop`，避免与普通 UI 框混用。
- 音效需要区分 UI、战斗、陷阱和桌面环境，便于低打扰模式统一控制音量。

### 4.5 `docs_runtime` 运行时文本目录

`docs_runtime` 用于存放可能被游戏直接读取的文本资产，例如战报模板、随机命名池、冒险者台词和事件短句。该目录与 `Docs` 不同，`Docs` 是项目文档，`docs_runtime` 是游戏内容资产。

```text
docs_runtime/
├─ names/
│  ├─ adventurer_names_zh.txt
│  ├─ monster_names_zh.txt
│  └─ title_pool_zh.txt
├─ reports/
│  ├─ battle_report_lines_zh.txt
│  └─ offline_summary_lines_zh.txt
└─ barks/
   ├─ adventurer_barks_zh.txt
   └─ monster_barks_zh.txt
```

运行时文本原则：

- 游戏内文本资产以中文为主，必要字段可保留英文 `id`。
- 模板文本应使用明确占位符，例如 `{adventurer_name}`、`{monster_name}`、`{gold}`。
- 随机文本需要按事件类型归档，避免所有文本堆在单个文件中。

### 4.6 `tools` 工具目录

`tools` 用于存放编辑器脚本、数据转换脚本、检查工具和临时开发辅助工具。

```text
tools/
├─ editor/
│  ├─ room_layout_editor/          # 房间布局编辑工具预留
│  └─ data_validator/              # 数据校验工具预留
├─ importers/
│  ├─ csv_to_resources/            # 表格转 Resource 工具预留
│  └─ text_template_checker/       # 文本模板检查工具预留
└─ debug/
   ├─ spawn_test_sets/             # 调试生成配置
   └─ balance_sandbox/             # 数值沙盒预留
```

工具目录原则：

- 工具脚本不应依赖正式运行场景才能执行。
- 数据转换结果进入 `data`，原始表格可按需要进入 `tools/importers` 或外部策划目录。
- 验证类工具应优先检查 `id` 重复、资源缺字段、模板占位符错误和引用路径失效。

### 4.7 `tests` 测试目录

`tests` 用于存放人工验证场景和后续自动化测试。桌面常驻项目存在较多窗口兼容风险，因此人工验证场景很重要。

```text
tests/
├─ manual/
│  ├─ desktop_window_test.tscn     # 贴边、透明、置顶、穿透测试
│  ├─ combat_loop_test.tscn        # 自动战斗闭环测试
│  ├─ offline_report_test.tscn     # 离线战报测试
│  └─ ui_layout_test.tscn          # UI 多分辨率测试
└─ unit/
   ├─ test_weighted_table.gd
   ├─ test_text_template.gd
   └─ test_economy_rules.gd
```

测试目录原则：

- `manual` 面向人工检查，尤其是桌面窗口、点击穿透、多显示器和高 DPI。
- `unit` 面向纯逻辑测试，优先覆盖文本模板、权重随机、经济规则和离线结算。
- 测试资源不应被正式发布场景直接依赖。

## 6. Autoload 建议

Godot 的 Autoload 应保持克制，仅用于跨场景长期存在的全局服务。推荐预留以下全局节点：

| Autoload | 建议路径 | 职责 |
|---|---|---|
| `Game` | `scripts/autoload/game.gd` | 游戏状态入口、版本状态和全局流程协调 |
| `SignalBus` | `scripts/autoload/signal_bus.gd` | 全局事件转发，降低模块直接依赖 |
| `DataRegistry` | `scripts/autoload/data_registry.gd` | 加载并索引 `data` 中的配置资源 |
| `SaveManager` | `scripts/autoload/save_manager.gd` | 存档、读档、版本迁移和异常处理 |
| `AudioManager` | `scripts/autoload/audio_manager.gd` | 音乐、音效、低打扰音量策略 |
| `DesktopWindowManager` | `scripts/autoload/desktop_window_manager.gd` | 桌面窗口模式、贴边、置顶、收起和兼容策略 |

不建议把魔物逻辑、冒险者逻辑、战斗细节或 UI 面板状态直接做成全局单例。

## 7. 资源命名规范

### 7.1 通用命名

- 文件和目录统一使用小写英文加下划线，例如 `battle_report_panel.tscn`。
- Godot 类名使用大驼峰，例如 `BattleReportPanel`。
- 数据 `id` 使用小写英文加下划线，例如 `slime_basic`、`trap_poison_fog`。
- 中文显示名只放在数据字段中，例如 `display_name = "史莱姆"`。
- 同一对象的场景、脚本、数据和贴图应尽量使用相同核心名，方便搜索。

### 7.2 场景命名

```text
desktop_root.tscn
rift_window.tscn
dungeon_root.tscn
room_slot.tscn
monster_slime.tscn
adventurer_rogue.tscn
battle_report_panel.tscn
```


## 8. 数据与资源引用原则

### 8.1 推荐引用方式

- 玩法对象通过数据 `id` 建立关系，避免大量硬编码路径。
- `DataRegistry` 统一加载 `data` 下的配置资源，并提供查询接口。
- 场景中可以导出 `Resource` 引用，但核心系统应支持通过 `id` 查找数据。
- 战报文本使用模板和占位符，事件系统只传入结构化结果。

### 8.2 不推荐方式

- 不推荐在战斗逻辑中直接写死某个魔物或冒险者名称。
- 不推荐 UI 面板直接修改数据资源原件。
- 不推荐把经济数值散落在不同脚本常量中。
- 不推荐在场景节点路径中承担业务关系，例如通过节点名字判断职业类型。

## 9. 版本阶段目录使用重点

### 9.1 VS 版本

重点建立 `scenes/desktop`、`assets/art/desktop`、`scenes/dungeon` 和少量 `assets/art/characters`，用于验证桌面裂隙视觉和基础观看体验。

### 9.2 FVS 版本

重点补充 `scripts/gameplay`、`scenes/entities` 和最小 `data` 内容，让冒险者进入、防守响应和结算反馈形成一条可观察链路。

### 9.3 MVP 版本

重点补齐 `data/monsters`、`data/adventurers`、`data/traps`、`data/resources`、`scenes/ui` 和 `scripts/save`，形成建设、防守、收益、升级和战报闭环。

### 9.4 Alpha 版本

重点扩展数据资产、事件资产、魔物成长、冒险者性格、复仇事件和窗口稳定性测试场景，目录结构不应再频繁大改。

### 9.5 CBT 版本

重点加强 `tests/manual`、`docs_runtime`、`tools/debug` 和反馈分析相关辅助资源，用于外部玩家封闭测试。

### 9.6 Beta 版本

重点完善 `assets`、`data`、`tests` 和兼容性验证资源，确保内容池、战报系统、存档和窗口行为接近公开版本质量。

### 9.7 EA 版本

重点维护可持续扩展的数据结构、内容包组织方式、更新文档和玩家反馈驱动的配置调整流程。

### 9.8 上线版本

重点整理正式发布资产、清理测试资源引用、确认商店素材、成就数据、版本说明和长期更新预留目录。

---

## 10. 禁止与注意事项

- 不要把正式项目文档散放到项目根目录，正式文档必须进入 `Docs` 对应子目录。
- 不要把游戏运行时文本资产放入 `Docs`，应放入 `docs_runtime` 或 `data/events`。
- 不要把临时测试场景混入正式场景目录，测试内容应进入 `tests`。
- 不要在根目录直接堆放美术、音频、表格和脚本文件。
- 不要让 `autoload` 承担过多玩法细节，避免后期难以测试和替换。
- 不要在窗口兼容性未验证前大量制作正式内容，应优先保证桌面常驻核心形态成立。

---

## 11. 推荐落地顺序

目录结构建议按版本逐步建立，不需要一次性创建所有空目录。

1. 先建立 `scenes/desktop`、`scripts/desktop`、`assets/art/desktop`，验证桌面窗口形态。
2. 再建立 `scenes/dungeon`、`scripts/dungeon`、`scenes/entities`，验证地下城和基础单位表现。
3. 然后建立 `data`、`scripts/resources`、`scripts/gameplay`，形成数据驱动的自动防守链路。
4. 接着建立 `scenes/ui`、`scripts/ui`、`docs_runtime`，补齐管理界面和战报反馈。
5. 最后完善 `tests`、`tools`、`assets/audio` 和更多内容资产，进入稳定迭代阶段。

---

## 12. 总结

《桌面裂隙》的目录资产结构应围绕“桌面窗口形态、地下城经营、防守自动化、事件战报、长期扩展”五个核心方向设计。

项目早期最重要的是保证目录边界清晰：桌面窗口归桌面模块，玩法规则归系统模块，内容配置归数据目录，表现资产归资产目录，项目文档归 `Docs`。只要从 VS 和 FVS 阶段开始保持这种结构，后续扩展到 MVP、Alpha、CBT、Beta、EA 和上线版本时，就能降低重构成本并提升内容生产效率。
